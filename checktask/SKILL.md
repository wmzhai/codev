# CheckTask

检查任务文件、更新通过项，并在全部通过时归档任务。

只做验证和状态更新，不主动修复代码问题。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T04` 或 `T04-foo.md`，使用该任务。
- 否则使用 `tasks/` 下编号最小的待办任务文件，忽略 `tasks/done/` 和 `tasks/README.md`。

## Workflow

1. 定位仓库根目录和目标任务文件。如果没有待办任务，明确告知所有任务已完成并停止。
2. 读取任务文件，提取 `## Acceptance Criteria` 或 `## 验收标准` 下的 checkbox 条目，兼容 bullet 和 numbered checkbox。
3. 逐项验证验收标准，优先使用最小侵入的方法：
   - 代码或配置类标准：读文件、搜索实现、核对结构。
   - 运行时标准：只在标准明确要求时运行针对性的命令或测试。
   - 含糊标准：标记为需要人工确认，不猜测通过。
4. 将明确通过的条目从 `[ ]` 更新为 `[x]`，失败或不确定的条目保持不变。
5. 如果存在未通过或待人工确认的条目：
   - 保留任务文件在原位置。
   - 向用户报告剩余缺口，不主动修复。
   - 仍然继续执行最后的代码精简审查建议。
6. 如果所有条目都通过：
   - 对比任务描述与实际实现，必要时仅同步那些已经和实现不一致的描述、字段名、方案说明或约束，保持文件结构不变。
   - 确保 `tasks/done/` 存在。
   - 在 git 仓库内优先使用 `git mv` 归档到 `tasks/done/`，否则使用普通移动。
7. 在结果中附带一次针对本次相关改动的精简审查建议：
   - 优先建议用户显式运行 `$simplify`。
   - 如果用户明确要求连同代码审查一起完成，再对相关 diff 做一次语义不变的简化检查。
8. 返回简洁结果块，包含任务名、每条标准的状态、总通过数，以及任务是否已归档。

## Rules

- 不要提交、推送或发版。
- 不要静默运行大而全的 CI；只运行验证当前标准所需的最小命令。
- 如果用户要求 review 风格输出，先给 findings，再给汇总。
- 如果验证命令因环境限制失败，要明确说明限制，不要把它当作产品缺陷。
- 不要为了“让任务通过”而修改验收标准本身；只同步那些已被实现事实取代的任务描述。
