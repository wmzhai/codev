# 安全、调试与专项审查

这份文档覆盖总流程中的非默认旁支：当任务涉及高风险命令、根因排查、性能或安全时，应该怎样插入对应工具，而不是把它们误当成默认主线的一部分。

## 何时进入这条旁支

- 你准备执行可能有破坏性的命令或生产操作。
- 用户报告了 bug，需要先做根因调查。
- 你要做性能回归、专项安全审计，或验证上线后的质量基线。

## 典型链路

```text
高风险操作前：
$careful / $freeze / $guard
-> 实施变更
-> $unfreeze

故障排查：
$investigate
-> 定位根因
-> 回到主流程实现修复

专项审查：
$benchmark
$cso
```

## 工具分工

- `$careful`
  - 对破坏性命令做警告，不限制编辑范围。
- `$freeze`
  - 把编辑范围锁到指定目录，避免顺手改别处。
- `$guard`
  - 组合启用 `$careful + $freeze`。
- `$unfreeze`
  - 清除 `$freeze` 或 `$guard` 的编辑边界。
- `$investigate`
  - 先调查、再假设、再修复；不接受“先拍脑袋改”。
- `$benchmark`
  - 记录或比较页面性能、资源体积和 Web Vitals。
- `$cso`
  - 做供应链、基础设施、OWASP/STRIDE 与 LLM 边界的安全审查。

## 使用规则

- 这些 skill 都是条件性旁支，不是每个 task 默认要跑。
- 调试流程中先跑 `$investigate`，再决定是否进入正常修复与 QA。
- `$benchmark` 与 `$cso` 的结果通常回流到 task plan、验证链或发布门禁。
