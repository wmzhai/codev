# PlanTask

围绕任务文件和现有代码，产出一份可以直接实现的计划。

本轮规划阶段只做研究、分析和决策，不编辑代码；方案输出后主动进入一个轻量确认步骤。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T05` 或 `T05-foo.md`，使用该任务。
- 否则使用 `tasks/` 下任务号最小的待办任务文件，只识别文件名匹配 `^T\\d+-.*\\.md$` 的任务文件，并忽略 `tasks/done/` 与其他非任务文件。

## Workflow

1. 定位仓库根目录并按上述顺序选择目标任务文件。如果没有待办任务，明确告知并停止。
2. 基于目标任务确定工作分支：
   - 取目标任务文件去掉 `.md` 后的文件名，作为该任务的工作分支名。
3. 检查当前 git 分支：
   - 如果当前分支不是 `main` 或 `master`，继续后续步骤。
   - 如果当前分支是 `main` 或 `master`，在开始规划前先切到上一步得到的任务分支。
   - 如果该分支已存在，直接切过去；如果不存在，就基于当前 `HEAD` 新建该分支并切过去。
   - 这一步只切分支，不做 commit、迁移或部署。
4. 读取整个任务文件，提取：
   - 目标和背景
   - 验收标准
   - 依赖任务
   - 约束、备注和相关代码入口
5. 检查依赖任务是否已完成；如果前置任务仍未完成，明确指出并停止，不绕过缺失前提继续规划。
6. 从任务文件提供的上下文出发深入代码：
   - 先快速搜索，再精读真正会改到的文件。
   - 优先复用现有模式、组件、工具函数和数据流。
   - 需要并行探索时可以使用 explorer agent。
7. 默认直接收敛实现方向，而不是把方案卡在中途确认：
   - 如果存在会显著改变实现的歧义，先尽量依据代码现状、任务约束和已有模式做保守判断，并在方案里写明假设。
   - 如果有多种可行路径，直接推荐一种并给出简短理由，把放弃其他路径的原因写进方案。
   - 只有在缺少关键外部事实、无法安全做出实现决策时，才向用户提一个最小问题。
   - 讨论重点放在实现层面的决策，不回到需求本身重复发散。
8. 输出一份可以立即实施的计划，至少包含：

```markdown
# Txx: 任务标题 - 实现方案

## Context

## Assumptions and Open Questions

## Files to Create or Modify
| File | Change | Reason |
|------|--------|--------|

## Implementation Details
### 1. 第一步
### 2. 第二步

## Execution Order
1. ...
2. ...

## Risks

## Validation
1. ...
```

9. 计划完成后，不要在本轮继续改代码；在回复结尾主动询问用户下一步：
   - 如果用户接受方案或明确表示按此执行，则下一轮直接进入实现，不要求用户重复描述要改什么。
   - 如果用户想继续交流、调整范围或修正方案，则继续停留在规划讨论阶段。

## Rules

- 规划粒度要到实现可落地，而不是泛泛而谈的架构建议。
- 优先写清楚假设和已确认决策，少写模糊措辞。
- 如果任务文件和代码现状冲突，以代码现状和用户最新确认为准，并在计划中显式写明。
- 识别任务编号时只兼容纯整数编号，例如 `T18`，不再使用字母后缀子任务。
- `tasks/` 中任务文件名本身就是索引和排序依据。
- 如果运行时还停留在 `main/master`，锁定任务后应先切到与该任务文件同名的分支，再继续规划和后续实现。
- 默认一次性给出可执行方案，不把“等待进一步确认”当成常规输出。
- 方案结尾必须主动给用户一个简短确认口：接受后开始实现，或继续讨论；不要要求用户手工重新下达“开始写代码”的完整指令。
- 不做文件编辑、提交、迁移或部署。
