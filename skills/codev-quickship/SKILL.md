# QuickShip

`codev-quickship` 是人工验证通过后的统一收尾 skill。它基于当前 task 的已确认结果，补齐最终 task 记录、归档任务、同步任务相关文档，并在仓库已有版本工件时做最小同步，然后完成“提交并推主干”：如果当前在分支上，就先把分支收成稳定状态再合并进主干；如果已经在主干上，就直接在主干上提交并推送。它不走标准 PR / release 链路，也不调用 `$ship`、`$land-and-deploy`、`$document-release`。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的检查、阻塞、合并结果和推送结果默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Preconditions

只有在以下前提满足时才继续：

- 用户显式调用 `$codev-quickship` 本身，就视为已经确认功能由人工验证通过，并接受当前结果进入收尾与 push 主干；除非用户在同轮明确撤回这两个前提。
- 当前仓库允许直接推送主干；如果有 branch protection、权限限制或必须走 PR 的规则，要明确说明并停止。
- 能明确定位本次要收尾的 task 文件；如果当前分支和用户输入都无法定位 task，不要猜测。
- 如果当前不在主干，当前分支必须处于可合并状态；如果存在未解决冲突或明显未收敛的工作区，先处理后再继续。

如果这些前提不成立，要明确说明原因并停止，不要擅自收尾或直推主干。

## Inputs

从用户请求与当前仓库状态推断目标：

- 默认以当前 git 分支作为工作起点。
- 目标分支优先使用仓库默认主干；如果无法判断，则按 `main` 优先、`master` 兜底。
- task 文件优先使用用户指定任务；否则优先匹配与当前分支同名的 `tasks/Txx-*.md`；仍无法确定时再按最小编号待办任务兜底。

## Workflow

1. 确认当前分支、目标主干与用户输入没有显式否定前提：
   - 调用 `$codev-quickship` 默认就表示“人工验证已经通过，且可以收尾并推主干”。
   - 只有当用户在同轮明确表示“尚未人工验证”或“不要 push 主干”时，才停止。
2. 定位目标 task，并完整读取当前任务文件：
   - 确认 task 文档与当前分支、当前实现是一致的。
   - 如果 task 无法明确对应到当前工作状态，直接停止并说明阻塞。
3. 检查当前工作区：
   - 如果有未提交改动，但它们显然属于本次 quickship，可先在当前分支做一次最小提交再继续。
   - 如果改动混杂、来源不清或存在高风险文件，不要替用户猜，直接停止并说明需要先整理工作区。
4. 先同步最终 task 文档：
   - 写入人工验证已经完成且通过的事实。
   - 更新 `Acceptance Criteria` / `验收标准` 中已确认通过的 checkbox。
   - 补齐最终实现、人工验证摘要、剩余风险、后续无需再做的待办项，以及适合归档的最终状态。
5. 归档 task：
   - 确保 `tasks/done/` 存在。
   - 在 git 仓库内优先使用 `git mv` 把任务文件移动到 `tasks/done/`。
6. 同步任务相关文档：
   - 只更新与本任务直接相关、且已经被人工验证确认的 `docs/`、`memory/`。
   - 如本任务改变了 repo 的稳定工作流、约束或默认动作，最小范围更新 `AGENTS.md`。
   - 如果没有需要更新的内容，要明确记录“任务相关 docs/memory/AGENTS 无需更新”。
7. 同步仓库里已存在的版本工件：
   - 只处理当前仓库已经存在的版本号文件或 `CHANGELOG`。
   - 如果仓库已有固定版本工件，按仓库既有规则做最小同步。
   - 如果仓库没有这些工件，要明确记录“已跳过版本更新”，不要自动新建文件。
8. 根据当前所在分支决定 quickship 路径：
   - 如果当前不在主干：先在当前分支提交本次收尾改动，再同步本地主干到最新远端状态，切到目标主干，把当前分支合并进主干；能 fast-forward 就保持简单线性历史，不能 fast-forward 时再做普通 merge。
   - 如果当前已经在主干：直接在当前主干上把本次收尾改动整理成一次最小提交。
9. 直接推送主干：
   - 把本地主干推到远端主干。
   - 如果推送失败、被保护规则拦截或权限不足，明确报告并停止，不要改走其他发布技能。
10. 用中文向用户汇报：
   - 目标 task 与归档位置
   - 起始分支
   - 目标主干
   - 走的是“branch merge”还是“main commit + push”
   - 是否已完成 task 归档
   - docs / memory / AGENTS 的更新结果
   - 版本号 / `CHANGELOG` 的同步结果，或明确说明已跳过
   - 是否已成功完成 commit / merge / push
   - 如果失败，卡在哪个 git / 权限步骤

## Rules

- `codev-quickship` 默认是人工验证通过后的收尾入口。
- 显式调用 `$codev-quickship` 就视为用户已经确认“人工验证通过 + 可以直接收尾并 push 主干”；不要重复追问同义确认问题。
- 不要调用 `$ship`、`$land-and-deploy`、`$document-release`。
- 不要创建 PR。
- 只处理仓库里已经存在的版本号文件或 `CHANGELOG`；如果不存在，要明确跳过，不自动新建。
- 不打 tag，也不做正式发布。
- 不做自动化功能验证；功能验证以用户明确完成的人工验证为准。
- `docs/`、`memory/`、`AGENTS.md` 的更新必须是任务相关、事实已确认、范围最小的同步，不要借机做全仓重写。
- 如果主干 push 之后会触发仓库自己的自动部署，这是仓库现状；`codev-quickship` 不负责等待部署或做 canary 验证。
- 如果用户需要 tag 或正式发布链路，应该转去 gstack 正常发布链路。

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 任务文件与归档位置
- 起始分支
- 目标分支
- 本次走的是 branch merge 还是 main commit + push
- docs / memory / AGENTS 的更新结果
- 版本号 / `CHANGELOG` 的同步结果
- 是否已成功完成 commit / merge
- 是否已成功 push 主干
- 是否已完成 task 归档
- 如果未完成，具体阻塞点
