# PlanTask

围绕任务文件和现有代码，产出一份可以直接实现的计划。

规划阶段只做研究、分析和决策，不编辑代码。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T05` 或 `T05-foo.md`，使用该任务。
- 否则使用 `tasks/` 下编号最小的待办任务文件，忽略 `tasks/done/` 和 `tasks/README.md`。

## Workflow

1. 定位仓库根目录和目标任务文件。如果没有待办任务，明确告知并停止。
2. 读取整个任务文件，提取：
   - 目标和背景
   - 验收标准
   - 依赖任务
   - 约束、备注和相关代码入口
3. 检查依赖任务是否已完成；如果前置任务仍未完成，明确指出并停止，不绕过缺失前提继续规划。
4. 从任务文件提供的上下文出发深入代码：
   - 先快速搜索，再精读真正会改到的文件。
   - 优先复用现有模式、组件、工具函数和数据流。
   - 需要并行探索时可以使用 explorer agent。
5. 先和用户对齐实现方向，而不是闷头给方案：
   - 如果存在会显著改变实现的歧义，先问清楚。
   - 如果有多种可行路径，推荐一种并给出简短理由，再和用户确认。
   - 讨论重点放在实现层面的决策，不回到需求本身重复发散。
6. 输出一份可以立即实施的计划，至少包含：

```markdown
# Txx: 任务标题 - 实现方案

## Context

## Assumptions and Open Questions

## Files to Create or Modify
| File | Change | Reason |
|------|--------|--------|

## Implementation Details
### 1. 第一步
### 2. 第二步

## Execution Order
1. ...
2. ...

## Risks

## Validation
1. ...
```

7. 计划完成后停止，不要继续改代码，除非用户明确要求开始实现。

## Rules

- 规划粒度要到实现可落地，而不是泛泛而谈的架构建议。
- 优先写清楚假设和已确认决策，少写模糊措辞。
- 如果任务文件和代码现状冲突，以代码现状和用户最新确认为准，并在计划中显式写明。
- 不做文件编辑、提交、迁移或部署。
